# Contributing to Mimer

Thanks for helping out. Mimer is a small, focused, MIT-licensed macOS app — issues
and PRs are welcome.

## Build & run

The Xcode project is **generated by [XcodeGen](https://github.com/yonaskolb/XcodeGen)
from `project.yml`** — the `.xcodeproj` and `Sources/Mimer/Info.plist` are git-ignored.
**Run `xcodegen generate` first**, or a fresh checkout won't build (and an already-open
project won't see new files):

```sh
brew install xcodegen          # one-time
git clone https://github.com/hasanjafri/Mimer.git
cd Mimer
xcodegen generate              # writes Mimer.xcodeproj from project.yml — do this after every pull that adds files
open Mimer.xcodeproj           # ⌘R to run
```

Requires **macOS 14+** and the Xcode command-line tools.

## Test

```sh
xcodegen generate
xcodebuild -project Mimer.xcodeproj -scheme Mimer -configuration Debug \
  -destination 'platform=macOS,arch=arm64' CODE_SIGNING_ALLOWED=NO test
```

CI runs the same build + test (and a Release build) on every PR; both must pass before merge.
Add a test with every behavior change — a regression test for a bug, both branches for a new
conditional. See `TESTING.md`/`CLAUDE.md` for the self-test loop used to exercise the UI.

## Conventions

- **Small, atomic commits** — one logical change each. Build + test before committing.
- Never hand-edit the generated `.xcodeproj` or `Info.plist`; change `project.yml` and re-run `xcodegen generate`.
- Dependencies are **exact-pinned** in `project.yml` (`exactVersion:`). Keep them pinned.
- Match the surrounding code's style and comment density.
- Update `README.md` / `CLAUDE.md` / `CHANGELOG.md` when your change affects them.

## Submitting a PR

1. Branch off `main` (it's protected — no direct pushes).
2. Make the change + tests; ensure `xcodebuild … test` passes locally.
3. Open the PR against `main` and fill in the template. CI must be green to merge.

## Architecture

`CLAUDE.md` is the fastest tour of the codebase (monitor → store → palette → paster,
encryption, the build/test/release flow). `docs/` has the deeper research, design, and reviews.

## Reporting bugs / security issues

File a bug with the issue template. For anything security-sensitive (this app handles
clipboard contents), follow [`SECURITY.md`](SECURITY.md) instead of opening a public issue.
